#+title: mttm — Multi-Tool Team Mate
#+subtitle: A reusable multi-agent configuration for OpenCode

This is my personal [[https://opencode.ai/][OpenCode]] agent+skill configuration. It provides a
multi-agent orchestration system that routes development tasks to specialized
subagents — each equipped with domain expertise, focused prompts, and
least-privilege file permissions.

* Overview

This project turns OpenCode into a coordinated team of specialists:

- **Orchestrator** – understands the full request, decomposes it, delegates
  subtasks to the right subagent, and integrates their outputs.
- **7 subagents** – domain experts for Rust, Haskell, Python, data
  engineering, reasoning/constraint solving, web development, and documentation.
- **11 skills** – on-demand workflow guidance loaded by agents when a task
  matches the skill's domain.
- **Nix flake support** – reproducible development environments for every
  language stack (environment definitions via [[https://nixos.wiki/wiki/Flakes][Nix flakes]]).

The project is designed to be:

- **Layered** – the orchestrator shields you from having to think about
  which specialist to call; you describe what you want, it handles the rest.
- **Extensible** – adding a new subagent or skill takes one file and one
  line in the orchestrator's prompt.
- **Reproducible** – Nix flakes pin toolchains so every agent gets the same
  environment every time.

** How it works

1. You send a request to OpenCode (e.g., "Build a CLI tool in Rust that
   queries DuckDB and outputs JSON").
2. The **orchestrator** receives it, analyzes the domains involved (Rust,
   data engineering), and decomposes it into subtasks.
3. Each subtask is dispatched to the appropriate **subagent** via the
   =task= tool. Subagents are isolated: they only see the context the
   orchestrator provides and can only edit files matching their
   permission patterns.
4. During execution, a subagent may load a **skill** (via the =skill=
   tool) for domain-specific workflows, templates, and conventions.
5. Results flow back to the orchestrator, which integrates them,
   resolves cross-domain conflicts, and presents the final output.

* Installation

** Prerequisites

- [[https://opencode.ai/][OpenCode]] installed and configured.
- Git.

** Steps

#+begin_src fish
# Clone the repository
git clone https://github.com/Dietr1ch/mttm ~/Projects/mttm/

# Symlink agents and skills into OpenCode's config directory
pushd ~/.config/opencode/
ln -s ~/Projects/mttm/agent
ln -s ~/Projects/mttm/skills
popd
#+end_src

That's it. OpenCode will pick up the orchestrator as the default agent
and all subagents will be available.

** Verify the installation

#+begin_src bash
~/Projects/mttm/scripts/validate-config.bash
#+end_src

This checks that:
- All agent files have valid YAML frontmatter.
- Every skill directory name matches its =name= field.
- Agent model assignments are consistent with the main config.

* Model Strategy

This project uses a two-tier model assignment to balance quality, cost,
and latency:

| Role                     | Model                          | Rationale                                                    |
|--------------------------+--------------------------------+--------------------------------------------------------------|
| Orchestrator             | =deepseek/deepseek-v4-flash=   | Strong reasoning for decomposition; moderate speed            |
| Rust subagent            | =ollama/qwen3-coder=           | Narrow, well-scoped Rust tasks; fast local inference          |
| Python subagent          | =ollama/qwen3-coder=           | Same rationale — local, fast, good enough for Python          |
| Web subagent             | =ollama/qwen3-coder=           | Same — vanilla JS/CSS/HTML is well within its capability      |
| Documentation subagent   | =deepseek/deepseek-v4-flash=   | Strong language understanding for clear, structured docs      |
| Haskell subagent         | =deepseek/deepseek-v4-flash=   | Haskell's type system benefits from stronger reasoning        |
| Data Engineering         | =deepseek/deepseek-v4-flash=   | Schema design and query optimization need deeper analysis     |
| Reasoning/Constraint     | =deepseek/deepseek-v4-pro-thinking=     | Prolog/MiniZinc require the strongest reasoning model         |

** When to use which model

- **=deepseek/deepseek-v4-flash=** — default workhorse. Good balance of
  reasoning depth and response speed. Used by the orchestrator (needs to
  plan and integrate) and by subagents whose domains benefit from deeper
  reasoning (Haskell types, SQL optimization).
- **=deepseek/deepseek-v4-pro-thinking=** — heavy reasoning. Used only by the
  reasoning-engineer subagent for Prolog, MiniZinc, SAT/SMT, and
  constraint satisfaction where correctness depends on precise logical
  inference.
- **=ollama/qwen3-coder=** — small, fast local model. Used for subagents
  with narrow, well-defined tasks (Rust, Python, web). The orchestrator
  provides enough context that the subagent only needs to generate
  straightforward code.

** Overriding models

To use a different model, edit =opencode.jsonc=:

#+begin_src jsonc
{
  "model": "your/preferred-model",       // Orchestrator + Haskell/Data/Reasoning
  "small_model": "your/small-model",      // Rust/Python/Web subagents
}
// Then update individual agent .md files' `model:` frontmatter if needed.
// The orchestrator's `model:` is set in its frontmatter and defaults to `deepseek-v4-flash`.
// Override with environment variable OPENCODE_MODEL or per-request model parameter.
#+end_src

** Orchestrator

The orchestrator is the entry point for every conversation. Its
responsibilities:

1. **Understand** the user's request and identify the domains involved.
2. **Decompose** into well-scoped subtasks, one per domain.
3. **Delegate** each subtask to the right subagent via the =task= tool.
4. **Integrate** the results into a coherent response, resolving any
   cross-domain conflicts (e.g., ensuring a Rust subagent's struct
   definitions match a data-engineer's SQL schema).

For simple or cross-domain tasks, the orchestrator may handle them
directly without delegation.

** Subagents

Each subagent has:
- A focused **prompt** describing its expertise and conventions.
- **Permission patterns** limiting file edits to relevant extensions
  (e.g., rust-dev can edit =*.rs= and =Cargo.toml=, but nothing else
  without asking).
- **Model** tuned to the domain's reasoning requirements.
- Access to relevant **skills** for domain-specific workflows.

* Customization

This configuration is designed to be forked and tailored. Here are the
common extension points.

** Adding a new subagent

1. Create =agent/<name>.md=:

   #+begin_src markdown
   ---
   description: What this agent does.
   mode: subagent
   model: <model-name>
   permission:
     edit:
       "**/*.<ext>": allow
       "*": ask
   ---

   You are a <domain> specialist. ...

   - Specific guidelines
   - Preferred libraries
   - Coding conventions
   #+end_src

2. Register the agent in =agent/orchestrator.md= by adding it to the
   "Available subagents" list.

** Adding a new skill

1. Create =skills/<name>/SKILL.md=:

   #+begin_src markdown
   ---
   name: <name>
   description: When to use this skill (appears in the agent's skill list).
   ---

   # <Skill Name>

   ## Workflow
   Step-by-step guidance for the domain.

   ## Conventions
   - Specific rules the agent should follow.
   #+end_src

2. The skill is automatically discoverable by OpenCode. Agents load it
   via the =skill= tool when the task matches its description.

** Modifying permissions

Each subagent's frontmatter includes a =permission.edit= block that
restricts which file patterns the agent can edit without asking. Follow
the principle of least privilege: only allow the extensions the agent
actually needs.

** Using a different model provider

Swap the model strings in =opencode.jsonc= and the agent frontmatter.
If you switch entirely to local models (e.g., all agents use Ollama),
remove the =small_model= key and use a single =model= for consistency.

* Development

** Validating the configuration

Run the validation script to catch common issues:

#+begin_src bash
./scripts/validate-config.bash
#+end_src

This checks:
- YAML frontmatter in all agent and skill files.
- Skill directory names match their =name= frontmatter field.
- Model consistency between =opencode.jsonc= and agent files.

** Conventions for this repo

- Agent files use YAML frontmatter with =description=, =mode=, =model=,
  and =permission= fields.
- Skills live in =skills/<name>/SKILL.md= with =name= and =description=
  in frontmatter. The =name= must match the directory name.
- Templates go in =skills/<name>/templates/=.
- The orchestrator =model= field must match =opencode.jsonc='s =model=.
- Subagents using the "small" tier should have =model= matching
  =opencode.jsonc='s =small_model=.

** Goals

- Offer agents that excel at multi-agent orchestration of tasks.
- Offer global skills helpful for single and multi-agent tasks.
- Use selected technologies:
  - Nix flakes for environment definition.
  - Rust for most tasks (high level, great performance, good libraries
    and type system).
  - Haskell (excellent type system, good performance).
  - Python (high level, vast library ecosystem).
  - Postgres/DuckDB for data handling.
  - Prolog and solvers like MiniZinc for reasoning.
  - HTML+vanilla JS for web pages and apps.
